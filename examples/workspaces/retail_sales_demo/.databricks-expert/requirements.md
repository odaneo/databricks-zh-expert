# 项目需求

## 业务目标

为 AWS 零售销售 Mock 项目建立统一分析数据基础，支持经营团队按日期、门店、商品、客户和渠道查看销售与库存表现。成功标准是口径可追溯、批流数据可关联，并且所有未确认设计都明确标记为提案。

## 源系统

主数据和库存快照来自 Amazon RDS for PostgreSQL。门店 POS 每小时向 S3 写入 Parquet 销售明细。电商订单、支付和用户行为事件进入 Amazon Kinesis Data Streams。

## 期望数据产品

需要四类数据产品：每日销售、商品表现、库存健康、客户与渠道分析。这里仅描述业务期望，不预定义 Databricks 目标表名或目标字段。

## 摄取需求

RDS 基准链路固定为 RDS PostgreSQL → AWS DMS → S3 Parquet → Auto Loader，先完成全量装载，再持续处理 CDC 文件。POS Parquet 使用 Auto Loader 增量摄取。Kinesis 事件使用正式可用的流式能力持续摄取，不采用 Preview 功能。

## 数据量与 SLA 假设

Mock 基线假设每日一千万条 POS 明细、五百万条电商事件和每小时一次库存快照。每日销售在次日 06:00 UTC 前可用，实时事件端到端延迟目标为五分钟；真实容量、峰值和恢复目标待确认。

## 治理与安全

客户姓名、邮箱、电话和地址属于直接识别信息。分析产品默认只发布不可逆客户标识和经批准的分群属性，并保留源文件、批次、事件时间和处理时间等审计信息。

## 技术约束

云平台固定为 AWS，目标平台为 Databricks。设计只能采用正式可用能力，不使用 Preview 功能；Agent 不连接 AWS 或 Databricks，不执行代码，也不在该输入目录写入生成结果。

## 待确认事项

需要业务和平台团队确认销售净额、退货、取消订单、库存缺货、客户归属渠道、时区、迟到边界、数据保留期、峰值吞吐和成本预算。
