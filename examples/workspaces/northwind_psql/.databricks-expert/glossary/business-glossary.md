# 业务词汇表

## 文档定位

本文是 Northwind 销售分析项目的统一术语入口，适用于需求、架构、业务规则、数据产品、数据质量、治理、运维、模型设计和评审材料。

每个术语至少明确以下内容：

- 规范中文名称
- 规范英文名称
- 业务含义
- 业务粒度
- 来源事实或计算依据
- 允许同义词
- 禁止解释和使用边界

本文中的英文名称用于项目沟通和逻辑设计。后续物理字段命名可以采用符合工程规范的英文标识，但含义必须与本文一致。

术语被列入本文只表示定义已经确认，不表示相关数据已经摄取、计算、认证、发布或通过质量检查。

## 术语使用规则

### 规范名称

同一概念在跨文档标题、指标目录、数据字典和评审材料中优先使用规范中文名称。

允许同义词可用于自然语言说明。涉及产品合同、质量规则、指标映射和验收时，应同时标明规范名称，避免将相近概念混用。

源字段名只用于追溯事实来源。源字段名不得替代业务术语，也不得扩展出源 Schema 未提供的业务语义。

### 指标定义完整性

销售指标必须同时明确以下要素：

- 业务粒度
- 认证销售范围
- 销售归属日期
- 交易单价来源
- 数量来源
- 折扣计算顺序
- 运费是否包含
- 定点小数类型和舍入规则
- 空值与未归属处理

缺少任一关键要素时，不得仅凭指标名称推断计算口径。

### 日期术语

订单创建日期、要求发货日期、实际发货日期、销售日期和快照日期是不同概念。文档和模型不得使用含义不明确的订单日期或业务日期替代这些规范术语。

Northwind 源日期均为日粒度，不含时分秒和时区。平台调度采用 `Asia/Tokyo`，运行审计时间采用 UTC。平台时区规则不得改变源日期值。

### 金额术语

所有金额统一解释为源系统金额，规范标签为 `source_currency_amount`。

源 Schema 没有货币代码。任何金额术语都不声明美元、日元或其他具体币种，也不包含汇率换算含义。

认证金额计算前必须将源 `real` 值转换为定点小数，并按 `HALF_UP` 规则保留四位小数。

## 订单

### 规范名称

| 项目 | 定义 |
|---|---|
| 规范中文名称 | 订单 |
| 规范英文名称 | Order |
| 来源对象 | `orders` |
| 稳定标识 | `order_id` |
| 业务粒度 | 每个 `order_id` 一张订单 |

### 业务定义

订单表示 Northwind 源系统中的一张订单头记录，承载客户、员工、订单创建日期、要求发货日期、实际发货日期、承运商、运费和允许摄取的配送地理信息。

`order_id` 是订单主键且不得为空。目标层必须完整保留源值，不得用自动生成编号替换该源业务键。

一张订单可以关联多条订单明细。订单数必须按 `order_id` 去重计算，不能按订单明细行数计算。

### 允许关系

订单通过源 DDL 中的外键关联以下实体：

- 客户通过 `customer_id`
- 员工通过 `employee_id`
- 承运商通过 `ship_via`
- 订单明细通过 `order_id`

`customer_id`、`employee_id` 和 `ship_via` 允许为空。合法空值保持为空，不创建未知客户、未知员工或未知承运商。

### 使用边界

源 Schema 没有统一订单状态、取消状态、付款状态、退货状态或退款状态。

未发货订单只能称为开放订单。不得将其自动解释为取消订单、失败订单、未付款订单或零销售订单。

`shipped_date` 非空只证明源系统记录了发货日期，不能证明客户已经收货。

## 订单明细

### 规范名称

| 项目 | 定义 |
|---|---|
| 规范中文名称 | 订单明细 |
| 规范英文名称 | Order Detail |
| 来源对象 | `order_details` |
| 组合主键 | `order_id` 加 `product_id` |
| 业务粒度 | 一张订单中的一个商品 |

### 业务定义

订单明细表示订单中的一个商品交易项。

`order_id` 和 `product_id` 共同组成组合主键，两个字段均不得为空。同一订单中的同一商品在源表中最多出现一行。

商品购买数量通过 `quantity` 表达。不得通过复制订单明细行表示数量。

订单明细提供历史销售计算所需的以下事实：

- 交易单价 `unit_price`
- 商品数量 `quantity`
- 折扣率 `discount`

### 使用边界

订单明细行数不能替代商品数量。

订单明细交易单价不能被商品当前目录单价覆盖。

订单明细没有独立发货日期。其销售日期继承所属订单的 `shipped_date`。

## 已发货订单、开放订单与认证销售

### 已发货订单

已发货订单的规范英文名称为 Shipped Order。

`orders.shipped_date` 非空时，订单属于已发货订单范围。

已发货订单可以成为认证销售候选。订单仍需通过适用的关键质量规则，并且至少包含一条通过质量规则的订单明细，才能进入认证销售。

已发货订单不等同于已送达订单。源 Schema 没有实际送达日期或签收结果。

### 开放订单

开放订单的规范英文名称为 Open Order。

`orders.shipped_date` 为空时，订单属于开放订单范围。

开放订单不进入每日销售、客户价值、商品与品类销售或员工销售表现。开放订单继续进入配送表现，并在 `order_date` 非空时计算开放订单天数。

开放订单不承载取消、失败或付款状态含义。

### 认证销售

认证销售的规范英文名称为 Certified Sales。

认证销售同时满足以下条件：

- `orders.shipped_date` 非空
- 订单通过适用的关键质量规则
- 订单至少包含一条通过质量规则的订单明细
- 订单明细通过源业务键正确关联订单

认证销售按实际发货日期归属。被隔离的订单或订单明细不进入受影响产品的认证结果。

认证销售表示通过业务和质量门禁的计算范围。该术语不能单独证明某次真实运行已经完成认证。

## 销售日期

### 规范名称

| 项目 | 定义 |
|---|---|
| 规范中文名称 | 销售日期 |
| 规范英文名称 | Sales Date |
| 来源字段 | `orders.shipped_date` |
| 数据粒度 | 日 |

### 业务定义

销售日期直接取 `orders.shipped_date`。

每日销售、客户价值累计、商品与品类日销售和员工销售表现统一使用销售日期归属认证销售。

`orders.order_date` 只表示订单创建日期，用于订单创建分析、发货处理天数和开放订单天数。

`orders.required_date` 只用于发货处理代理指标。

### 使用边界

不得使用订单创建日期替代销售日期。

不得根据 DMS 提交时间、文件修改时间、摄取时间或 Job 运行时间重写销售日期。

销售日期不含时分秒和时区。平台调度时区只决定日终认证边界，不改变源日期。

## 交易单价与当前目录单价

### 交易单价

交易单价的规范英文名称为 Transaction Unit Price。

交易单价直接来自 `order_details.unit_price`，表示订单明细中的历史成交价格，是销售金额计算的唯一单价来源。

认证计算前将交易单价转换为 `DECIMAL 18,4`。

交易单价必须大于等于零。负值属于关键质量失败。

### 当前目录单价

当前目录单价的规范英文名称为 Current Catalog Unit Price。

当前目录单价直接来自 `products.unit_price`，表示商品当前状态中的目录价格，只用于商品库存观察和当前商品属性分析。

当前目录单价不得回填、覆盖或重算历史订单明细金额。

交易单价与当前目录单价不同属于允许的业务现象，可生成数据质量警告，但不构成历史交易价格错误。

## 商品数量

商品数量的规范英文名称为 Item Quantity。

订单行商品数量直接取 `order_details.quantity`。该值必须大于零。

跨订单、日期、客户、商品、品类或员工汇总时，商品数量等于通过质量规则的 `quantity` 之和。

订单明细行数、去重商品数和商品数量是不同指标：

- 订单明细行数表示交易项数量
- 去重商品数表示不同 `product_id` 的数量
- 商品数量表示 `quantity` 的合计值

## 折扣率

折扣率的规范英文名称为 Discount Rate。

折扣率直接来自 `order_details.discount`。认证计算前转换为 `DECIMAL 9,6`。

有效范围为零到一，包含两个边界。

- 零表示无折扣
- 一表示订单行净销售额为零
- 小于零或大于一属于关键质量失败

异常折扣率不得裁剪为边界值，也不得静默改写为零。

## 毛销售额

### 规范术语关系

| 使用层级 | 规范中文名称 | 规范英文名称 |
|---|---|---|
| 订单明细层级 | 订单行毛额 | Gross Sales Amount at Order Detail Grain |
| 聚合指标层级 | 商品毛额 | Gross Sales Amount |
| 允许同义词 | 毛销售额 | Gross Sales Amount |

项目指标目录优先使用商品毛额。自然语言中可以使用毛销售额，但必须保持相同计算含义。

### 订单行毛额

订单行毛额的计算规则为：

`订单行毛额 = 交易单价 × 商品数量`

计算顺序如下：

1. 将 `order_details.unit_price` 转换为 `DECIMAL 18,4`
2. 使用通过质量规则的 `quantity`
3. 执行乘法
4. 按 `HALF_UP` 保留四位小数

### 商品毛额

商品毛额等于认证范围内已舍入订单行毛额之和。

每日销售、客户价值、商品与品类表现和员工销售表现都从已舍入订单行毛额汇总，不能先汇总源浮点数再统一舍入。

### 使用边界

商品毛额不扣除折扣。

商品毛额不包含订单运费。

商品毛额不包含取消、退货或退款调整。源 Schema 没有这些事实。

## 折扣金额

### 规范术语关系

| 使用层级 | 规范中文名称 | 规范英文名称 |
|---|---|---|
| 订单明细层级 | 订单行折扣额 | Discount Amount at Order Detail Grain |
| 聚合指标层级 | 折扣额 | Discount Amount |
| 允许同义词 | 折扣金额 | Discount Amount |

项目指标目录优先使用折扣额。折扣金额可以作为自然语言同义词。

### 订单行折扣额

订单行折扣额的计算规则为：

`订单行折扣额 = 已舍入订单行毛额 × 折扣率`

计算顺序如下：

1. 使用已按 `HALF_UP` 保留四位小数的订单行毛额
2. 使用转换为 `DECIMAL 9,6` 的折扣率
3. 执行乘法
4. 按 `HALF_UP` 保留四位小数

### 折扣额

折扣额等于认证范围内已舍入订单行折扣额之和。

折扣额是金额指标。折扣率是比例指标。两个术语不得互换。

### 使用边界

折扣额不包含运费优惠、退款、返利、优惠券或税费调整。源 Schema 没有这些事实。

## 净销售额

### 规范术语关系

| 使用层级 | 规范中文名称 | 规范英文名称 |
|---|---|---|
| 订单明细层级 | 订单行净销售额 | Net Sales Amount at Order Detail Grain |
| 订单层级 | 订单商品净销售额 | Order Net Sales Amount Excluding Freight |
| 聚合指标层级 | 商品净销售额 | Net Sales Amount |
| 允许同义词 | 净销售额 | Net Sales Amount |

项目指标目录优先使用商品净销售额。净销售额可以作为自然语言同义词，但必须明确不含运费。

### 订单行净销售额

订单行净销售额的计算规则为：

`订单行净销售额 = 已舍入订单行毛额 − 已舍入订单行折扣额`

结果保留四位小数。

### 订单商品净销售额

订单商品净销售额等于同一 `order_id` 下全部已舍入订单行净销售额之和。

订单商品净销售额不包含 `orders.freight`。

### 商品净销售额

商品净销售额等于认证范围内已舍入订单行净销售额之和。

每日销售、客户价值、商品与品类表现和员工销售表现中的净销售额都从相同订单行净销售额汇总。

### 使用边界

商品净销售额不包含运费。

商品净销售额不包含税费。源 Schema 没有税费字段。

商品净销售额不扣除退货、退款或取消金额。源 Schema 没有相关事实。

商品净销售额只统计认证销售范围内的已发货订单与有效订单明细。

## 运费金额与含运费订单总额

### 运费金额

运费金额的规范英文名称为 Freight Amount。

运费直接来自 `orders.freight`，业务粒度为订单。

认证计算前将运费转换为 `DECIMAL 18,4`。

运费为空时，指定金额指标按零计入，并保留运费缺失标识或缺失计数。Silver 原始空值继续保留。

运费小于零属于关键质量失败。

同一订单的运费只能累计一次。订单与多条订单明细连接后不得重复累计运费。

### 含运费订单总额

含运费订单总额的规范英文名称为 Order Total Including Freight。

计算规则为：

`含运费订单总额 = 订单商品净销售额 + 订单运费`

跨订单汇总时，先在订单粒度确定商品净销售额和运费，再执行聚合。

### 产品使用范围

运费可以进入以下数据产品：

- 每日销售
- 客户价值
- 配送表现中的订单运费字段

运费不分摊到以下销售产品：

- 商品与品类销售
- 员工销售表现

### 平均商品净订单金额

平均商品净订单金额的规范英文名称为 Average Net Sales per Shipped Order。

计算规则为：

`平均商品净订单金额 = 商品净销售额 ÷ 已发货订单数`

该指标不包含运费。分子和分母必须来自相同认证范围和相同业务切片。

已发货订单数为零时，该指标保持为空，不得以零代替或执行除零计算。

含义不明确的平均订单金额应改用平均商品净订单金额，或明确说明是否包含运费。

## 客户价值

### 规范名称

| 项目 | 定义 |
|---|---|
| 规范中文名称 | 客户价值 |
| 规范英文名称 | Customer Value |
| 业务粒度 | 每个快照日期和 `customer_id` 一条记录 |
| 时间范围 | 从最早可用销售日期累计至快照日期 |
| 销售范围 | 可归属于客户的认证销售 |

### 业务定义

客户价值表示截至快照日期，每个当前客户已经产生的可观察历史销售贡献。

客户全集来自 Silver 当前客户状态。没有已发货订单的当前客户仍保留一条快照记录，金额和计数为零，平均商品净订单金额和销售日期相关指标为空。

只有 `orders.customer_id` 非空且能够引用有效客户的认证销售才能归属于客户价值。

`customer_id` 为空的已发货订单可以进入每日销售、商品与品类表现和适用的员工销售表现，但不进入任何客户价值记录，也不创建未知客户。

### 客户价值组成

客户价值可以包含以下已确认指标：

- 已发货订单数
- 商品数量
- 商品毛额
- 折扣额
- 商品净销售额
- 运费金额
- 运费缺失订单数
- 平均商品净订单金额
- 首次销售日期
- 最近销售日期
- 最近购买间隔
- 去重商品数
- 去重品类数

首次销售日期和最近销售日期都基于 `orders.shipped_date`。

最近购买间隔等于快照日期减最近销售日期。没有已发货订单时，首次销售日期、最近销售日期和最近购买间隔保持为空。

### 当前属性边界

客户公司名、城市、地区和国家使用 Silver 当前属性。

源 Schema 没有客户属性有效期。历史客户价值刷新后可以显示最新当前属性，客户稳定归属继续使用 `customer_id`。

### 使用边界

客户价值只表达已观察历史贡献。

客户价值不表示预测客户终身价值、未来收入、流失概率、营销评分或客户等级。

当前客户价值不使用 `customer_demographics` 或 `customer_customer_demo` 计算指标。

## 延期发货

### 规范名称

| 项目 | 定义 |
|---|---|
| 规范中文名称 | 延期发货代理状态 |
| 允许业务简称 | 延期发货 |
| 规范英文名称 | Shipped After Required Date Proxy |
| 判断字段 | `orders.shipped_date` 和 `orders.required_date` |
| 业务粒度 | 订单 |

### 判断规则

同时满足以下条件时，延期发货代理状态为 true：

- `shipped_date` 非空
- `required_date` 非空
- `shipped_date` 晚于 `required_date`

`shipped_date` 小于等于 `required_date` 时，延期发货代理状态为 false，并进入按要求日期发货代理范围。

任一所需日期为空时，延期发货代理状态保持为空，不得推断 true 或 false。

### 晚于要求日期天数

晚于要求日期天数的规范英文名称为 Days Shipped After Required Date。

- 延期发货代理状态为 true 时，记录 `shipped_date` 减 `required_date` 的正数天数
- 按要求日期或提前发货时，记录零
- 未发货或要求日期为空时，保持为空

### 业务边界

延期发货代理状态只描述发货处理结果，不代表实际延迟送达。

该状态不能用于声明准时送达率、实际运输时长、客户签收结果或承运商最终配送绩效。

延期发货代理状态属于可分析业务结果，不构成数据质量失败。

`shipped_date` 早于 `order_date` 或 `required_date` 早于 `order_date` 属于日期一致性关键质量失败，与延期发货业务状态分开处理。

## 配送相关术语

### 发货处理天数

发货处理天数的规范英文名称为 Days to Ship。

计算规则为：

`发货处理天数 = 实际发货日期 − 订单创建日期`

只有 `shipped_date` 和 `order_date` 都非空时计算。任一字段为空时保持为空。

该指标描述从订单创建到源系统记录发货的日数，不表示运输时长。

### 按要求日期发货代理标识

按要求日期发货代理标识的规范英文名称为 Shipped by Required Date Proxy。

只有 `shipped_date` 和 `required_date` 都非空时计算：

- `shipped_date` 小于等于 `required_date` 时为 true
- `shipped_date` 晚于 `required_date` 时为 false
- 任一日期为空时保持为空

该指标只描述发货处理结果，不能命名为准时送达标识。

### 开放订单天数

开放订单天数的规范英文名称为 Open Order Days。

适用于 `shipped_date` 为空的订单。

计算规则为：

`开放订单天数 = 当前认证日期 − 订单创建日期`

`order_date` 为空时保持为空。

开放订单天数描述订单在源系统中尚未记录发货日期的累计日数，不表示取消、失败或客户等待责任。

## 数据产品

### 规范名称

| 项目 | 定义 |
|---|---|
| 规范中文名称 | 数据产品 |
| 规范英文名称 | Data Product |
| 项目范围 | 五类逻辑分析交付物 |

### 业务定义

数据产品是面向明确业务问题的受治理逻辑交付物，具有已确认的业务粒度、维度、指标、事实边界、质量门禁、访问规则、刷新要求、发布状态和责任主体。

数据产品定义属于业务合同。物理 Catalog、Schema、表、视图、文件、Notebook 和工作流属于后续实现设计，不能反向改变数据产品含义。

### 五类数据产品

| 规范中文名称 | 规范英文名称 | 业务粒度 | 核心范围 |
|---|---|---|---|
| 每日销售 | Daily Sales | 每个销售日期一条记录 | 按 `shipped_date` 归属的认证销售汇总 |
| 客户价值 | Customer Value | 每个快照日期和客户一条记录 | 当前客户截至快照日期的已观察历史贡献 |
| 商品与品类表现 | Product and Category Performance | 每个销售日期和商品，以及每个快照日期和商品 | 已发货商品销售与当前库存观察 |
| 员工销售表现 | Employee Sales Performance | 每个销售日期和员工一条记录 | 按 `orders.employee_id` 归属的认证销售 |
| 配送表现 | Shipping Performance | 每个订单一条当前记录 | 已发货与未发货订单的发货处理代理分析 |

### 每日销售

每日销售按销售日期聚合认证销售。

核心指标包括已发货订单数、去重客户数、商品数量、商品毛额、折扣额、商品净销售额、运费金额、含运费订单总额、平均商品净订单金额和运费缺失订单数。

每日销售不生成没有已发货销售的日期行。

### 客户价值产品

客户价值产品覆盖 Silver 当前客户全集。

该产品按快照日期累计可归属客户的认证销售，并保留没有已发货订单的当前客户。

### 商品与品类表现产品

商品与品类表现包含两个逻辑数据集：

- 商品与品类日销售表现
- 商品库存观察快照

历史销售金额使用订单明细交易单价。库存观察使用商品当前目录价格、库存、在途、补货阈值和停售状态。

### 员工销售表现产品

员工销售表现只按 `orders.employee_id` 归属订单。

员工销售区域关系不能用于订单区域推断、销售分摊或业绩重复归属。

员工销售表现不分摊订单运费。

### 配送表现产品

配送表现同时覆盖已发货和未发货订单。

该产品可以描述订单创建、要求发货、实际发货、开放订单天数和承运商当前属性。

该产品不能发布实际送达日期、准时送达率、实际运输时长、客户签收结果或承运商最终配送绩效。

### 发布状态

数据产品使用以下发布状态：

| 状态 | 英文名称 | 含义 |
|---|---|---|
| 候选 | `candidate` | 结果已生成或正在验证，尚未满足全部认证发布条件 |
| 已认证 | `certified` | 适用关键质量规则、对账、新鲜度和发布门禁已经在对应运行证据中通过 |
| 已阻断 | `blocked` | 存在关键失败，受影响结果不能推进认证水位或覆盖最近一次已认证结果 |

文档中定义已认证状态不代表真实环境当前处于 `certified`。真实状态只能由运行记录和发布证据确认。

## 跨产品公共术语

### 业务粒度

业务粒度的规范英文名称为 Business Grain。

业务粒度说明一条记录代表的最小业务对象或汇总组合。任何指标定义都必须与其业务粒度一致。

核心粒度包括：

- 订单按 `order_id`
- 订单明细按 `order_id` 加 `product_id`
- 每日销售按销售日期
- 客户价值按快照日期加客户
- 商品日销售按销售日期加商品
- 商品库存观察按快照日期加商品
- 员工销售按销售日期加员工
- 配送表现按订单

### 源业务键

源业务键的规范英文名称为 Source Business Key。

源业务键直接来自 Northwind 主键或组合主键，用于识别业务对象、CDC 去重、删除和稳定关联。

目标层不得用自动生成键覆盖源业务键。代理键可以服务物理实现，但不能改变源键语义或写回源键字段。

### 源事实

源事实的规范英文名称为 Source Fact。

源结构事实指注册文件 `source-schema/northwind-schema.sql` 中已定义的表、字段、类型、主键、外键和可空性。样例验收事实从固定的 `upstream/northwind.sql` 独立计算，只用于非生产测试和验收，不属于发送给模型的源结构上下文。

DMS 操作元数据、文件路径、摄取时间、质量状态、发布水位和数据产品指标属于平台或派生信息，不属于 Northwind 源事实。

### 当前属性

当前属性的规范英文名称为 Current Attribute。

当前属性来自 Silver 当前状态，包括客户公司名、商品名、当前品类、当前供应商、员工姓名、员工职位、员工上级关系和承运商名称。

源 Schema 没有属性有效期。当前属性变化后，历史汇总重新发布时可以显示新属性。

### 稳定归属

稳定归属的规范英文名称为 Stable Attribution。

稳定归属使用源标识符确定业务归属，例如订单客户使用 `customer_id`，订单员工使用 `employee_id`，订单商品使用 `product_id`。

名称、职位、品类名或其他当前属性变化不得改变源标识归属。

源 Schema 无法支持订单发生时的历史客户属性、历史员工组织关系或历史商品分类还原。

### 未归属记录

未归属记录的规范英文名称为 Unattributed Record。

允许为空的外键保持为空时，记录可以保留在适用的总量产品中，但不能归属于对应实体产品。

典型规则包括：

- 空客户订单不进入客户价值
- 空员工订单不进入员工销售表现
- 空承运商订单在配送表现中保留空承运商属性
- 空品类或空供应商商品保留商品事实和空属性

未归属记录不能通过 Unknown、Unassigned、N A、零值标识符或人工生成实体键进行填充。

### 快照日期

快照日期的规范英文名称为 Snapshot Date。

快照日期表示累计或观察结果的业务截止日期。

- 客户价值快照使用前一东京日历日期
- 商品库存观察快照使用日终 Job 运行时的东京日历日期

商品库存快照表示 Pipeline 观察时点的当前状态，不能声明为源系统日终库存。

### 刷新时间

刷新时间的规范英文名称为 Refresh Time。

刷新时间表示数据产品最近一次成功刷新完成的 UTC 时间。

刷新时间属于平台审计信息，不能替代销售日期、订单创建日期、实际发货日期或快照日期。

### 认证发布水位

认证发布水位的规范英文名称为 Certified Publication Watermark。

认证发布水位表示最近一次通过适用质量、对账、新鲜度和发布门禁的处理边界。

失败运行和已阻断运行不得推进认证发布水位，也不得覆盖最近一次已认证结果。

### 迟到数据

迟到数据的规范英文名称为 Late Arriving Data。

正常 CDC 端到端新鲜度目标为 P95 在 15 分钟内，最大允许延迟为 30 分钟。

超过 30 分钟到达的数据仍需处理，并记录 SLA 违约和迟到原因。迟到数据不得因超过时限而被丢弃。

### 补数

补数的规范英文名称为 Backfill。

补数表示对历史日期、业务键或快照范围执行受控重算和确定性覆盖。

标准补数窗口为最近 90 天。超过 90 天的历史更正需要产品 Owner 和平台 Owner 双重批准。

补数不能无条件追加重复聚合，也不能推进未经质量和对账验证的认证水位。

### 数据代际

数据代际的规范英文名称为 Dataset Generation。

数据代际用于区分 Full Load 重建、受控重新初始化和持续 CDC 产生的不同数据版本体系。

数据代际属于平台处理语义，不属于 Northwind 源业务字段。

不同数据代际必须按已批准顺序参与版本选择，防止旧代际数据覆盖新代际当前状态。

### 关键质量失败

关键质量失败的规范英文名称为 Critical Quality Failure。

关键质量失败会隔离受影响记录，并阻断受影响数据产品推进认证发布。

典型情况包括主键缺失、持久外键孤儿、非法数量、负交易单价、非法折扣、负运费、日期顺序错误、Restricted 字段进入平台和认证数据存在未处理救援字段。

### 数据质量警告

数据质量警告的规范英文名称为 Data Quality Warning。

警告允许处理继续，但必须写入质量结果并由数据 Steward 处置。

典型情况包括可空客户或员工导致未归属、当前目录价格与历史交易价格不同、文本格式异常和当前属性变化导致历史展示重述。

### 隔离记录

隔离记录的规范英文名称为 Quarantined Record。

隔离记录保存因关键质量失败退出认证处理范围的数据及其可追溯证据。

隔离证据至少关联源业务键、规则标识、发现时间、来源文件和适用平台元数据。

隔离不允许静默修改源值。修复后必须重新通过质量检查、对账和发布门禁。

## 数据分类术语

### Restricted

Restricted 表示项目中最高限制等级的源字段。

联系人、电话、详细地址、个人出生信息、员工备注、图片、图片路径和订单详细收货信息属于 Restricted 范围。

Restricted 字段在 DMS 字段白名单阶段排除，不进入 S3、Databricks 或数据产品。

任何 Restricted 字段进入平台都属于 Sev 1 安全事件。

### Confidential

Confidential 表示需要按批准用途、最小权限、审计和列级保护发布的数据。

客户公司名、员工姓名、员工职位、员工入职日期、员工汇报关系、供应商公司名、承运商公司名，以及获准保留的城市、地区和国家字段属于 Confidential 范围。

普通授权分析师读取客户公司名和员工姓名时使用稳定别名。获批产品 Owner 和数据 Steward 可以按职责读取原值。

### Internal

Internal 表示仅供项目内部授权用途的数据和指标。

订单、订单明细、商品、品类、库存、销售区域、源标识符和认证聚合指标属于 Internal 范围。

### Public

Public 表示允许公开发布的数据。

当前项目没有源字段或数据产品被确认为 Public，不建立公共发布渠道。

## 同义词与禁止误用

| 使用表达 | 规范处理 |
|---|---|
| 毛销售额 | 允许作为商品毛额的同义词 |
| 订单行毛销售额 | 使用订单行毛额 |
| 折扣金额 | 允许作为折扣额的同义词 |
| 净销售额 | 允许作为商品净销售额的同义词，并明确不含运费 |
| 订单净销售额 | 明确使用订单商品净销售额，并说明不含运费 |
| 订单总额 | 含义不明确，改用订单商品净销售额或含运费订单总额 |
| 销售单价 | 含义不明确，改用交易单价或当前目录单价 |
| 订单日期 | 含义不明确，改用订单创建日期、销售日期或其他明确日期 |
| 发货日期 | 优先使用实际发货日期，字段追溯为 `shipped_date` |
| 延迟送达 | 当前项目禁止使用，源 Schema 没有实际送达日期 |
| 延期发货 | 允许作为延期发货代理状态的业务简称，使用时必须保留发货处理或代理说明 |
| 准时送达率 | 当前项目禁止使用，改用按要求日期发货代理指标 |
| 客户终身价值 | 当前项目禁止作为客户价值同义词，项目没有预测未来价值 |
| 已完成订单 | 当前项目不使用，源 Schema 没有统一完成状态 |
| 已送达订单 | 当前项目禁止使用，源 Schema 没有实际送达或签收事实 |
| 取消订单 | 当前项目不能从现有源字段推断 |
| 未分配客户 | 改用客户标识为空或客户未归属 |
| 未分配员工 | 改用员工标识为空或员工未归属 |
| Unknown 客户 | 禁止创建，保留空源键并通过未归属统计表达 |
| Unknown 员工 | 禁止创建，保留空源键并通过未归属统计表达 |
| 区域销售 | 当前项目不能依据员工销售区域推导订单区域销售 |
| 源系统日终库存 | 当前项目禁止使用，库存快照只表示 Pipeline 观察状态 |

## 引用关系

术语定义的权威引用顺序如下：

1. 源表、字段、主键、外键、类型和可空性引用规范 Northwind SQL 文件
2. 销售、折扣、日期、配送、空值和归属公式引用已确认业务与数据规则
3. 五类产品的业务问题、粒度、指标和交付边界引用数据产品定义
4. 质量失败、警告、隔离和发布门禁引用数据质量要求
5. 数据分类、访问控制、保留、SLA、Owner 和成本引用治理与运维要求
6. 摄取、分层、调度、重跑和数据代际引用项目架构与摄取约束

本文不能单独作为源 Schema、指标实现、质量执行结果或部署状态的证明。

## 变更与失效

新增术语必须明确规范中文名称、规范英文名称、定义、粒度、来源、允许同义词和禁止解释。

计算术语发生变化时，必须先更新已确认业务与数据规则，再同步更新数据产品定义、数据质量要求、模型设计、测试和本文。

源字段或关系发生变化时，必须先更新规范 Northwind SQL 文件，再评估受影响术语。

术语更名必须记录原名称、新名称、生效日期、兼容期和旧名称失效状态。

禁止仅在 SQL、Notebook、Dashboard 或模型回答中改变术语含义。

当前版本没有阻断性待确认术语。
