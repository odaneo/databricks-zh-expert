# 数据产品定义

## 文档定位

本文件定义 Northwind 销售分析项目的五类逻辑数据产品：

- 每日销售
- 客户价值
- 商品与品类表现
- 员工销售表现
- 配送表现

五类产品共同服务销售趋势、客户贡献、商品经营、员工归属和发货处理分析。产品只使用源 Schema 已定义并经字段最小化批准进入平台的数据。

本文中的产品定义属于业务合同。物理对象、存储结构、计算实现和发布代码由后续设计文档承接，并且必须保持与本文一致。

## 产品组合总览

| 数据产品 | 核心业务问题 | 业务粒度 | 主要时间基准 | 认证销售范围 |
|---|---|---|---|---|
| 每日销售 | 每个销售日期实现了多少订单、客户、商品数量和销售金额 | 每个销售日期一条汇总记录 | `orders.shipped_date` | 只包含已发货且通过关键质量规则的订单与明细 |
| 客户价值 | 截至快照日期，每个当前客户贡献了多少已观察历史销售 | 每个快照日期和客户一条记录 | 快照日期与 `orders.shipped_date` | 只包含归属于该客户的已发货且通过质量规则的订单与明细 |
| 商品与品类表现 | 每个商品和当前品类的日销售表现如何，当前库存观察状态如何 | 销售部分按销售日期和商品，库存部分按快照日期和商品 | `orders.shipped_date` 与库存观察日期 | 销售部分只包含已发货订单，库存部分覆盖当前商品状态 |
| 员工销售表现 | 每个员工在每个销售日期被归属了多少已发货销售 | 每个销售日期和员工一条记录 | `orders.shipped_date` | 只包含 `orders.employee_id` 可归属的已发货订单 |
| 配送表现 | 每张订单何时创建、要求何时发货、实际何时发货，以及开放订单积压多久 | 每个订单一条当前记录 | `order_date`、`required_date`、`shipped_date` | 同时覆盖已发货订单和未发货订单 |

所有产品的业务 Owner 固定为 `grp_northwind_product_owners`。数据定义、质量和分类由 `grp_northwind_stewards` 负责。平台运行由 `grp_northwind_platform` 负责。授权分析使用者为 `grp_northwind_analysts`。

## 共同事实边界

所有产品必须遵守以下事实边界：

- 认证销售只统计 `orders.shipped_date` 非空的订单。
- 销售归属日期直接使用 `orders.shipped_date`。
- `orders.order_date` 表示订单创建日期，不承担认证销售归属。
- `orders.required_date` 只用于发货处理代理指标。
- 销售金额使用 `order_details.unit_price`，不得使用 `products.unit_price` 回填历史交易价格。
- 运费来自 `orders.freight`，只能在订单粒度累计一次。
- 商品、品类和员工销售表现不分摊运费。
- 源金额没有货币代码，统一标记为源系统金额，不声明具体币种，不做汇率换算。
- 源 Schema 没有取消、付款、退货、退款和实际送达事实，产品不得生成相关指标或状态。
- 当前名称、品类、供应商、职位、经理关系和承运商名称来自 Silver 当前状态。
- 源 Schema 没有属性有效期，历史销售刷新后可以显示最新当前属性。
- 稳定归属依赖源标识符，名称变化不得改变订单、客户、商品和员工的源标识归属。
- Restricted 字段不进入 S3、Databricks 或任何数据产品。
- 产品不得创建未知客户、未知员工、未知商品、未知品类、未知供应商或未知承运商等伪造维度成员。

## 每日销售

### 产品目标

每日销售产品提供按发货日期认证的销售总览，用于观察已实现销售趋势、订单规模、客户覆盖、商品数量、折扣和运费。

该产品回答以下业务问题：

- 每个销售日期有多少张已发货订单
- 每个销售日期覆盖多少个可识别客户
- 每个销售日期售出多少件商品
- 每个销售日期产生多少商品毛额、折扣额和商品净销售额
- 每个销售日期产生多少订单运费
- 每个销售日期的含运费订单总额是多少
- 每个销售日期的平均商品净订单金额是多少
- 每个销售日期有多少订单缺少运费值

### 业务粒度

每个 `sales_date` 一条记录。

`sales_date` 直接取 `orders.shipped_date`。只有存在通过关键质量规则的已发货订单时才形成对应销售日期记录。产品不额外生成没有已发货销售的日期行。

### 覆盖范围

纳入范围：

- `orders.shipped_date` 非空
- 订单通过适用的关键质量规则
- 订单至少包含一条通过关键质量规则的订单明细
- 订单明细通过 `order_id` 正确关联订单

排除范围：

- `shipped_date` 为空的开放订单
- 被关键质量规则隔离的订单或订单明细
- 无法完成源业务键关联的记录

`customer_id` 或 `employee_id` 为空不会使有效订单退出每日销售。客户去重数只统计非空 `customer_id`。空员工标识不影响每日销售总额。

### 来源事实

| 来源 | 使用事实 |
|---|---|
| `orders` | `order_id`、`customer_id`、`employee_id`、`shipped_date`、`freight` |
| `order_details` | `order_id`、`product_id`、`unit_price`、`quantity`、`discount` |

### 维度

| 维度 | 定义 |
|---|---|
| 销售日期 | `orders.shipped_date` |

每日销售保持单一日期粒度，不在同一产品中加入客户、商品、员工、承运商或配送地区明细。

### 指标

| 指标 | 定义 |
|---|---|
| 已发货订单数 | 按 `order_id` 去重后的有效已发货订单数量 |
| 去重客户数 | 有效已发货订单中非空 `customer_id` 的去重数量 |
| 商品数量 | 通过质量规则的 `quantity` 之和 |
| 商品毛额 | 已舍入订单行毛额之和 |
| 折扣额 | 已舍入订单行折扣额之和 |
| 商品净销售额 | 已舍入订单行净销售额之和 |
| 运费金额 | 订单粒度运费之和，空值按零计入并保留缺失计数 |
| 含运费订单总额 | 商品净销售额加运费金额 |
| 平均商品净订单金额 | 商品净销售额除以已发货订单数，不含运费；已发货订单数为零时为 null |
| 运费缺失订单数 | `freight` 为空的有效已发货订单数量 |

### 产品边界

- 订单数和运费在同一订单中只计算一次。
- 商品数量使用 `quantity`，不得使用订单明细行数替代。
- 每日销售不提供取消销售额、退货销售额、退款金额或付款状态。
- 每日销售不提供按订单创建日期归属的认证销售指标。
- 每日销售不声明金额币种。

## 客户价值

### 产品目标

客户价值产品提供截至快照日期的客户已观察历史价值，用于识别客户贡献、购买频次、最近购买时间和商品覆盖。

该产品回答以下业务问题：

- 每个当前客户截至快照日期产生了多少张已发货订单
- 每个当前客户累计购买了多少件商品
- 每个当前客户累计产生多少商品毛额、折扣额、商品净销售额和运费
- 每个当前客户的平均商品净订单金额是多少
- 每个当前客户首次和最近一次已发货销售日期是什么
- 每个当前客户距离最近一次已发货销售有多少天
- 每个当前客户购买过多少个不同商品和当前品类
- 哪些当前客户尚未产生可归属的已发货销售

### 业务粒度

每个快照日期和 `customer_id` 一条记录。

客户集合以 Silver 当前客户状态为准。每个当前客户都保留记录，包括截至快照日期没有已发货订单的客户。

历史价值范围从最早可用的已发货销售日期累计至快照日期。销售归属使用 `orders.shipped_date`。

### 快照规则

- 日终认证快照日期使用前一东京日历日期。
- 只累计 `shipped_date` 小于等于快照日期的认证销售。
- 没有已发货订单的当前客户保留零金额和零计数。
- 没有已发货订单的客户平均商品净订单金额、首次销售日期、最近销售日期和最近购买间隔保持为空。
- 客户从 Silver 当前状态删除后，不进入后续当前客户快照。既有历史快照按保留策略继续受控保存。

### 覆盖范围

纳入范围：

- Silver 当前客户全集
- 能通过非空 `orders.customer_id` 归属于客户的认证销售

不归属范围：

- `orders.customer_id` 为空的已发货订单
- 经过宽限后仍无法引用有效客户的订单

空客户订单仍可进入每日销售、商品与品类表现和适用的员工销售表现，但不进入任何客户价值记录，也不创建未知客户。

### 来源事实

| 来源 | 使用事实 |
|---|---|
| `customers` | `customer_id`、`company_name`、`city`、`region`、`country` |
| `orders` | `order_id`、`customer_id`、`shipped_date`、`freight` |
| `order_details` | `order_id`、`product_id`、`unit_price`、`quantity`、`discount` |
| `products` | `product_id`、`category_id` |

### 维度与属性

| 维度或属性 | 定义 |
|---|---|
| 快照日期 | 产品观察和累计截止日期 |
| 客户标识 | `customers.customer_id` |
| 客户公司名 | `customers.company_name` 的当前值 |
| 客户城市 | `customers.city` 的当前值 |
| 客户地区 | `customers.region` 的当前值 |
| 客户国家 | `customers.country` 的当前值 |

客户公司名、城市、地区和国家属于 Confidential 数据，只能按批准用途发布。客户公司名对普通授权分析师使用基于 `customer_id` 的稳定别名。

### 指标

| 指标 | 定义 |
|---|---|
| 已发货订单数 | 归属于客户的有效已发货订单去重数 |
| 商品数量 | 归属于客户的有效订单明细 `quantity` 之和 |
| 商品毛额 | 归属于客户的已舍入订单行毛额之和 |
| 折扣额 | 归属于客户的已舍入订单行折扣额之和 |
| 商品净销售额 | 归属于客户的已舍入订单行净销售额之和 |
| 运费金额 | 归属于客户的订单粒度运费之和，空值按零计入 |
| 运费缺失订单数 | 归属于客户且 `freight` 为空的有效已发货订单数量 |
| 平均商品净订单金额 | 客户商品净销售额除以客户已发货订单数，不含运费；已发货订单数为零时为 null |
| 首次销售日期 | 客户最早 `shipped_date` |
| 最近销售日期 | 客户最晚 `shipped_date` |
| 最近购买间隔 | 快照日期减最近销售日期 |
| 去重商品数 | 客户购买过的不同 `product_id` 数量 |
| 去重品类数 | 客户购买商品当前 `category_id` 的非空去重数量 |

### 产品边界

- 客户价值表示已观察历史贡献，不提供预测客户终身价值。
- 产品不生成客户等级、流失概率、推荐分群或营销评分。
- 当前商品品类变化可以重述客户历史去重品类数。
- 源 Schema 没有客户属性有效期，客户公司名和地区属性使用当前值。
- 客户人口统计关联表不进入当前客户价值指标。

## 商品与品类表现

### 产品目标

商品与品类表现产品同时提供商品日销售表现和商品库存观察快照，用于分析商品贡献、折扣、当前品类归属、供应商归属、库存状态和补货提示。

该产品由两个逻辑数据集组成：

- 商品与品类日销售表现
- 商品库存观察快照

两个逻辑数据集共享 `product_id`，但时间语义和事实来源不同。历史交易金额来自订单明细，当前目录价格和库存来自商品当前状态。

### 商品与品类日销售表现

#### 业务问题

- 每个销售日期每个商品售出了多少件
- 每个销售日期每个商品覆盖多少订单和可识别客户
- 每个销售日期每个商品产生多少商品毛额、折扣额和商品净销售额
- 每个商品当前归属于哪个品类和供应商
- 当前品类维度下的商品销售贡献如何

#### 业务粒度

每个销售日期和 `product_id` 一条记录。

只为存在有效已发货销售的商品和日期组合生成记录。品类汇总由商品记录按当前 `category_id` 聚合获得。

#### 覆盖范围

- 只包含通过关键质量规则的已发货订单与订单明细。
- 商品标识来自 `order_details.product_id`。
- `category_id` 或 `supplier_id` 为空时保持为空。
- `customer_id` 为空的订单仍进入商品销售金额和订单数，客户去重数不统计空值。
- 订单员工是否为空不影响商品销售归属。

#### 来源事实

| 来源 | 使用事实 |
|---|---|
| `orders` | `order_id`、`customer_id`、`shipped_date` |
| `order_details` | `order_id`、`product_id`、`unit_price`、`quantity`、`discount` |
| `products` | `product_id`、`product_name`、`category_id`、`supplier_id` |
| `categories` | `category_id`、`category_name` |
| `suppliers` | `supplier_id`、`company_name` |

#### 维度与属性

| 维度或属性 | 定义 |
|---|---|
| 销售日期 | `orders.shipped_date` |
| 商品标识 | `order_details.product_id` |
| 商品名称 | `products.product_name` 的当前值 |
| 品类标识 | `products.category_id` 的当前值 |
| 品类名称 | `categories.category_name` 的当前值 |
| 供应商标识 | `products.supplier_id` 的当前值 |
| 供应商公司名 | `suppliers.company_name` 的当前值 |

供应商公司名属于 Confidential 数据，只能向批准主体发布。

#### 指标

| 指标 | 定义 |
|---|---|
| 订单数 | 包含该商品的有效已发货订单去重数 |
| 去重客户数 | 购买该商品且 `customer_id` 非空的客户去重数 |
| 商品数量 | 该商品有效订单明细 `quantity` 之和 |
| 商品毛额 | 该商品已舍入订单行毛额之和 |
| 折扣额 | 该商品已舍入订单行折扣额之和 |
| 商品净销售额 | 该商品已舍入订单行净销售额之和 |

商品和品类销售不分摊订单运费。

### 商品库存观察快照

#### 业务问题

- 每个观察日期有哪些当前商品
- 每个商品当前目录价格是多少
- 每个商品当前库存、在途数量和可用供应数量是多少
- 每个商品当前补货阈值是多少
- 每个商品是否满足已确认补货提示规则
- 每个商品当前是否停售

#### 业务粒度

每个快照日期和 `product_id` 一条记录。

快照覆盖 Silver 当前商品全集，包括停售商品和没有历史销售的商品。

#### 快照时间

- 快照日期使用日终 Job 运行时的东京日历日期。
- 实际观察时间以 UTC 记录。
- 源商品表没有更新时间，快照表示 Pipeline 观察时点的当前状态。
- 快照不得声明为源系统日终库存。

#### 来源事实

| 来源 | 使用事实 |
|---|---|
| `products` | 全部已批准摄取的商品字段 |
| `categories` | `category_id`、`category_name` |
| `suppliers` | `supplier_id`、`company_name` |

#### 维度与属性

| 维度或属性 | 定义 |
|---|---|
| 快照日期 | 东京日历观察日期 |
| 观察时间 | Pipeline 实际观察 UTC 时间 |
| 商品标识 | `products.product_id` |
| 商品名称 | `products.product_name` |
| 品类标识 | `products.category_id` |
| 品类名称 | `categories.category_name` |
| 供应商标识 | `products.supplier_id` |
| 供应商公司名 | `suppliers.company_name` |
| 包装描述 | `products.quantity_per_unit` |

#### 指标与状态

| 指标或状态 | 定义 |
|---|---|
| 当前目录单价 | `products.unit_price` 的当前值，只表示目录价格 |
| 当前库存数量 | `products.units_in_stock` 的当前值 |
| 当前在途数量 | `products.units_on_order` 的当前值 |
| 可用供应数量 | 当前库存数量和当前在途数量分别将空值按零处理后相加 |
| 补货阈值 | `products.reorder_level` 的当前值 |
| 停售标识 | `products.discontinued` 为零时表示未停售，为一时表示已停售 |
| 补货标识 | 按已确认补货规则计算的三值结果 |

补货标识规则：

- 商品已停售时为 false。
- 商品未停售且补货阈值非空，并且可用供应数量小于等于补货阈值时为 true。
- 商品未停售且补货阈值非空，并且可用供应数量大于补货阈值时为 false。
- 补货阈值为空时保持为空。

`units_in_stock` 和 `units_on_order` 的源空值只在计算可用供应数量时按零处理，原字段空值状态必须保留。

### 产品边界

- 历史销售金额不得使用当前目录单价重算。
- 当前品类和供应商属性可能重述历史商品销售展示。
- 该产品不提供采购订单状态、供应商交付时间、库存成本或缺货损失。
- 该产品不推断源系统日终库存。
- `discontinued` 出现零和一以外的值时，受影响商品数据进入隔离并阻断发布。

## 员工销售表现

### 产品目标

员工销售表现产品按源订单员工标识归属已发货销售，用于观察员工日销售贡献、客户覆盖、商品数量、折扣和平均订单金额。

该产品回答以下业务问题：

- 每个销售日期每个员工被归属了多少张已发货订单
- 每个销售日期每个员工覆盖多少个可识别客户
- 每个销售日期每个员工销售多少件商品
- 每个销售日期每个员工产生多少商品毛额、折扣额和商品净销售额
- 每个销售日期每个员工的平均商品净订单金额是多少
- 每个员工当前职位和当前上级是谁

### 业务粒度

每个销售日期和 `employee_id` 一条记录。

只为存在至少一张可归属有效已发货订单的员工和日期组合生成记录。

### 归属规则

- 订单只按 `orders.employee_id` 归属员工。
- 一张订单最多归属一个员工。
- `orders.employee_id` 为空的已发货订单不进入任何员工销售记录。
- 空员工订单仍进入每日销售、客户价值和商品与品类表现中的适用结果。
- 空员工订单生成数据质量警告，不创建未知员工。
- `employee_territories`、`territories` 和 `region` 不用于订单销售区域推断或业绩重复分配。

### 来源事实

| 来源 | 使用事实 |
|---|---|
| `orders` | `order_id`、`customer_id`、`employee_id`、`shipped_date` |
| `order_details` | `order_id`、`unit_price`、`quantity`、`discount` |
| `employees` | `employee_id`、`last_name`、`first_name`、`title`、`reports_to` |

### 维度与属性

| 维度或属性 | 定义 |
|---|---|
| 销售日期 | `orders.shipped_date` |
| 员工标识 | `orders.employee_id` |
| 员工姓名 | 当前 `first_name` 和 `last_name` 组合展示 |
| 员工职位 | `employees.title` 的当前值 |
| 上级员工标识 | `employees.reports_to` 的当前值 |

员工姓名、职位和汇报关系属于 Confidential 数据。员工姓名对普通授权分析师使用基于 `employee_id` 的稳定别名。

### 指标

| 指标 | 定义 |
|---|---|
| 已发货订单数 | 归属于员工的有效已发货订单去重数 |
| 去重客户数 | 员工订单中非空 `customer_id` 的去重数量 |
| 商品数量 | 员工订单中有效明细 `quantity` 之和 |
| 商品毛额 | 员工订单中已舍入订单行毛额之和 |
| 折扣额 | 员工订单中已舍入订单行折扣额之和 |
| 商品净销售额 | 员工订单中已舍入订单行净销售额之和 |
| 平均商品净订单金额 | 员工商品净销售额除以员工已发货订单数，不含运费 |

员工销售表现不分摊运费。

### 产品边界

- 员工归属只反映 `orders.employee_id`，不代表佣金归属、团队归属或区域归属。
- 当前员工职位和上级关系可能重述历史销售展示。
- 源 Schema 没有历史组织有效期，产品不得声明订单发生时的历史经理关系。
- 产品不发布员工个人联系方式、出生日期、地址、家庭电话、备注、图片或图片路径。
- 产品不计算销售目标达成率，因为源 Schema 没有目标数据。

## 配送表现

### 产品目标

配送表现产品提供订单创建、要求发货、实际发货、开放订单账龄和承运商当前属性，用于分析订单发货处理过程。

该产品回答以下业务问题：

- 每张订单是否已发货
- 已发货订单从创建到发货经历多少天
- 已发货订单是否在要求日期之前或当天发货
- 晚于要求日期发货的订单晚了多少天
- 未发货订单截至当前认证日期开放了多少天
- 哪些订单缺少要求日期或运费
- 每张订单当前关联哪个承运商和配送城市、地区、国家

### 业务粒度

每个 `order_id` 一条当前记录。

产品同时覆盖已发货订单和未发货订单。源端删除的订单在 CDC 处理和后续发布后从当前产品中移除。

### 来源事实

| 来源 | 使用事实 |
|---|---|
| `orders` | `order_id`、`order_date`、`required_date`、`shipped_date`、`ship_via`、`freight`、`ship_city`、`ship_region`、`ship_country` |
| `shippers` | `shipper_id`、`company_name` |

### 维度与属性

| 维度或属性 | 定义 |
|---|---|
| 订单标识 | `orders.order_id` |
| 订单创建日期 | `orders.order_date` |
| 要求发货日期 | `orders.required_date` |
| 实际发货日期 | `orders.shipped_date` |
| 承运商标识 | `orders.ship_via` |
| 承运商公司名 | `shippers.company_name` 的当前值 |
| 配送城市 | `orders.ship_city` |
| 配送地区 | `orders.ship_region` |
| 配送国家 | `orders.ship_country` |

承运商公司名和保留的配送地理字段属于 Confidential 数据，只能向批准主体发布。

`ship_via` 为空时，承运商标识和名称保持为空，不创建未知承运商。

### 指标与状态

| 指标或状态 | 定义 |
|---|---|
| 运费金额 | `orders.freight`，空值按零用于金额并保留缺失标识 |
| 已发货标识 | `shipped_date` 非空时为 true，否则为 false |
| 发货处理天数 | `shipped_date` 减 `order_date`，两个日期都非空时计算 |
| 按要求日期发货代理标识 | `shipped_date` 和 `required_date` 都非空时计算，前者小于等于后者时为 true |
| 晚于要求日期天数 | 发货晚于要求日期时记录正数，按期或提前发货时记录零 |
| 开放订单天数 | 未发货订单使用当前认证日期减 `order_date` |
| 要求日期缺失标识 | `required_date` 为空时为 true |
| 运费缺失标识 | `freight` 为空时为 true |

空值规则：

- 未发货订单的发货处理天数保持为空。
- 未发货订单的按要求日期发货代理标识保持为空。
- 未发货订单的晚于要求日期天数保持为空。
- `order_date` 为空时，发货处理天数和开放订单天数保持为空。
- `required_date` 为空时，按要求日期发货代理标识和晚于要求日期天数保持为空。

### 代理指标边界

源 Schema 没有实际送达日期。`required_date` 和 `shipped_date` 的比较只描述源系统中的发货处理代理结果。

配送表现不得发布或暗示以下内容：

- 准时送达率
- 实际送达日期
- 实际运输时长
- 承运商最终配送绩效
- 客户签收结果
- 配送异常原因

按承运商进行分析时，结论只能描述订单记录中的发货处理代理分布，不得扩展为最终配送服务评价。

## 跨产品公共维度与指标

### 公共日期语义

| 日期概念 | 定义 | 使用产品 |
|---|---|---|
| 销售日期 | `orders.shipped_date` | 每日销售、客户价值、商品与品类销售、员工销售 |
| 订单创建日期 | `orders.order_date` | 配送表现 |
| 要求发货日期 | `orders.required_date` | 配送表现 |
| 快照日期 | 日终产品观察和累计截止日期 | 客户价值、商品库存观察快照 |
| 刷新时间 | 产品最近成功刷新 UTC 时间 | 全部产品 |

源日期都是 PostgreSQL `date`，不含时分秒和时区。目标逻辑保持日粒度，不对源日期执行时区换算。

平台调度时区为 `Asia/Tokyo`。运行审计、摄取时间和 DMS 提交时间使用 UTC。

### 公共标识符

| 实体 | 稳定源标识符 |
|---|---|
| 订单 | `order_id` |
| 客户 | `customer_id` |
| 员工 | `employee_id` |
| 商品 | `product_id` |
| 品类 | `category_id` |
| 供应商 | `supplier_id` |
| 承运商 | `shipper_id` |

所有跨产品关联优先使用稳定源标识符。名称只用于展示，不承担业务键职责。

### 当前属性原则

以下属性使用 Silver 当前状态：

- 客户公司名和客户地理属性
- 商品名称
- 商品当前品类
- 商品当前供应商
- 品类名称
- 供应商公司名
- 员工姓名
- 员工职位
- 员工上级关系
- 承运商公司名

源 Schema 没有历史有效期字段。当前属性发生变化后，历史汇总刷新时可以显示新属性。产品必须保留源标识符，使使用者能够区分标识归属和展示属性。

### 公共销售指标

| 指标 | 公共定义 |
|---|---|
| 订单行毛额 | 转换后的订单明细交易单价乘数量，并按 `HALF_UP` 保留四位小数 |
| 订单行折扣额 | 已舍入订单行毛额乘转换后的折扣，并按 `HALF_UP` 保留四位小数 |
| 订单行净销售额 | 已舍入订单行毛额减已舍入订单行折扣额 |
| 商品毛额 | 已舍入订单行毛额之和 |
| 折扣额 | 已舍入订单行折扣额之和 |
| 商品净销售额 | 已舍入订单行净销售额之和 |
| 商品数量 | 通过质量规则的 `quantity` 之和 |
| 已发货订单数 | 认证范围内按 `order_id` 去重的订单数量 |
| 去重客户数 | 认证范围内非空 `customer_id` 的去重数量 |
| 平均商品净订单金额 | 商品净销售额除以已发货订单数，不含运费 |

### 金额与舍入

认证计算前必须执行以下类型标准化：

| 源值或结果 | 认证类型 |
|---|---|
| `order_details.unit_price` | `DECIMAL 18,4` |
| `products.unit_price` | `DECIMAL 18,4` |
| `orders.freight` | `DECIMAL 18,4` |
| `order_details.discount` | `DECIMAL 9,6` |
| 派生金额 | `DECIMAL 20,4` |

金额计算顺序固定为：

1. 将交易单价、折扣和运费转换为定点小数。
2. 计算订单行毛额并按 `HALF_UP` 保留四位小数。
3. 使用已舍入订单行毛额计算折扣额并按 `HALF_UP` 保留四位小数。
4. 使用已舍入毛额减已舍入折扣额得到订单行净销售额。
5. 从已舍入订单行金额汇总订单、日期、客户、商品、品类和员工金额。
6. 所有认证金额保留四位小数。

认证计算不得直接使用 FLOAT 或 DOUBLE 聚合后再舍入。

### 运费原则

- 运费位于订单粒度。
- 运费为空时，金额按零计入，并保留缺失标识或缺失计数。
- 运费小于零时按关键质量失败处理。
- 每张订单的运费只能累计一次。
- 运费可以进入每日销售和客户价值。
- 运费不进入商品、品类和员工销售金额。
- 含运费订单总额等于商品净销售额加运费。

### 空值归属原则

| 空值情况 | 处置 |
|---|---|
| `orders.customer_id` 为空 | 订单保留在适用的总销售与商品结果中，不归属客户价值，客户去重数不计空值 |
| `orders.employee_id` 为空 | 订单保留在适用的其他销售结果中，不归属员工销售 |
| `orders.ship_via` 为空 | 配送记录保留，承运商属性为空 |
| `products.category_id` 为空 | 商品记录保留，品类属性为空 |
| `products.supplier_id` 为空 | 商品记录保留，供应商属性为空 |
| `orders.freight` 为空 | 金额按零计入并保留缺失状态 |
| 当前名称为空 | 保留源标识符和空名称，不生成替代名称 |

非空外键在 30 分钟宽限后仍无法引用有效父记录时，按关键质量失败处理。

### 敏感信息发布

Restricted 字段在源端字段白名单阶段排除，任何数据产品都不得包含以下内容：

- 联系人姓名和联系人职位
- 电话和传真
- 街道地址和邮政编码
- 员工出生日期和家庭电话
- 员工备注、图片和图片路径
- 订单收货人和详细收货地址
- 二进制图片内容

Confidential 字段按批准用途发布，并接受 Unity Catalog 最小权限、审计和列级保护。

- 客户公司名对普通授权分析师显示稳定别名。
- 员工姓名对普通授权分析师显示稳定别名。
- 产品 Owner 和数据 Steward 可以按职责查看批准的原值。
- 当前项目不实施基于组织边界的行级过滤，因为源 Schema 没有可验证的组织访问边界字段。

## 刷新与交付要求

### 增量刷新

数据产品由已确认的 RDS PostgreSQL、AWS DMS、S3 Parquet、Auto Loader、Bronze、Silver 和 Gold 链路提供。

增量编排要求：

- 由文件到达事件触发。
- 最小触发间隔为 5 分钟。
- 同一编排最多一个活动运行实例。
- Gold 对受影响日期、业务键或快照执行确定性覆盖。
- 失败、超时、质量阻断和对账失败不得推进认证水位。
- 源端删除在目标 15 分钟内从 Silver 当前状态和后续产品刷新中移除。

正常 CDC 端到端新鲜度从 DMS 源提交时间到产品最近刷新 UTC 时间计算。

| 服务指标 | 已确认目标 |
|---|---|
| 新鲜度 P95 | 15 分钟内 |
| 最大允许延迟 | 30 分钟 |
| 月度数据产品可用性 | 99.5 百分比 |

超过 30 分钟到达的数据仍必须处理，并记录 SLA 违约和迟到原因。

### 日终认证

日终认证在每日 01:30 按 `Asia/Tokyo` 运行，并在 02:00 前完成。

日终认证规则：

- 每日销售认证前一东京日历日期。
- 客户价值快照日期使用前一东京日历日期。
- 商品库存观察快照日期使用 Job 运行时的东京日历日期。
- 配送表现每日重算开放订单天数。
- 即使没有新文件，日终认证仍必须执行。
- 失败运行不得覆盖最近一次已认证结果。

### 发布状态

每个数据产品使用以下受控发布状态：

- `candidate`
- `certified`
- `blocked`

只有同时满足以下条件时，产品才能更新为 `certified`：

- 适用的关键质量规则全部通过
- 增量对账成功
- 数据新鲜度未超过 30 分钟
- 输入和输出水位完整记录
- 受影响产品计算成功

关键失败时，受影响产品更新为 `blocked`。最近一次已认证结果继续可用，失败运行不得将其覆盖。

### 质量依赖

产品发布至少依赖以下关键质量要求：

- 源主键非空
- Silver 主键和组合主键唯一
- 非空外键在宽限后可引用有效父记录
- `quantity` 大于零
- 订单明细交易单价大于等于零
- 折扣位于零到一之间
- 运费大于等于零或为空
- 已发货订单至少包含一条有效订单明细
- 发货日期不早于订单创建日期
- 要求发货日期不早于订单创建日期
- 库存、在途和补货阈值大于等于零或为空
- 停售值只能为零或一
- 金额完成定点小数转换且没有溢出
- Restricted 字段扫描结果为零
- 认证数据没有未处理的救援字段

质量警告允许处理继续，但必须写入质量结果并由数据 Steward 处置。任何质量规则都不得静默改写源值。

### 重跑与补数

- 标准重跑和补数窗口为最近 90 天。
- 超过 90 天的历史更正需要产品 Owner 和平台 Owner 双重批准。
- `shipped_date` 从空值变为日期时，订单进入认证销售。
- `shipped_date` 从日期变为空值时，订单从认证销售移除。
- `shipped_date` 改为其他日期时，原销售日期和新销售日期都要重算。
- 客户、员工、商品、价格、数量、折扣或运费更正时，重算受影响产品结果。
- 重跑使用确定性覆盖，禁止无条件追加重复聚合。

### 交付方式

- 数据产品通过受 Unity Catalog 管理的 Gold 数据提供。
- 交互式分析和 BI 查询通过生产 Serverless SQL Warehouse 提供。
- 产品 Owner、数据 Steward 和授权分析师按职责读取 Gold 产品。
- 普通分析师读取客户公司名和员工姓名时应用已确认 Column Mask。
- 分析师不直接读取 Landing、Bronze 或 Silver。
- 每个产品提供最近刷新 UTC 时间、认证状态和可追溯发布水位。
- 产品交付不得包含真实凭据、连接信息、Restricted 字段或未经批准的导出副本。

### 保留要求

| 数据产品内容 | 保留期 |
|---|---|
| 销售明细与聚合 | 7 年 |
| 商品库存观察快照 | 3 年 |
| 质量、对账和运行元数据 | 400 天 |
| 隔离记录 | 90 天 |

源端删除和具有法律效力的数据删除指令遵循项目数据保留与例外删除流程。

### 产品验收

每类产品进入认证发布前必须完成以下验收：

- 业务粒度与本文一致
- 源字段只来自批准的源 Schema 和摄取白名单
- 销售范围只包含已发货订单
- 销售日期使用 `shipped_date`
- 订单行金额符合定点小数和 `HALF_UP` 规则
- 运费没有因订单明细连接重复累计
- 商品、品类和员工销售没有分摊运费
- 客户和员工空值归属符合本文规则
- 当前属性和历史归属边界已向使用者说明
- 配送指标没有声明实际送达事实
- 关键质量规则和对账通过
- Restricted 字段扫描结果为零
- Confidential 字段的访问和列级保护通过验证
- 刷新时间、认证状态和水位可追溯

本文只定义验收要求，不据此声称任何环境已经完成验收或发布。

## 不在当前产品范围

以下能力不属于当前五类数据产品：

- 取消订单分析
- 退货和退款分析
- 付款和应收分析
- 币种识别和汇率换算
- 实际送达和签收分析
- 准时送达率和真实运输时长
- 承运商最终配送绩效
- 预测客户终身价值
- 客户流失预测和营销推荐
- 销售目标和达成率
- 员工佣金计算
- 历史组织关系还原
- 基于员工销售区域的订单归属
- 采购订单和供应商交付分析
- 源系统日终库存还原

新增以上能力前必须增加可验证的数据来源，并完成业务口径、源映射、质量、治理、历史兼容性和产品验收评审。
